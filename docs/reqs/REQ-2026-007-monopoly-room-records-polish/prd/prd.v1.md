# 大富翁地图与房间体验优化 (REQ-2026-007)

## 1. 背景与目标
- 当前大富翁对局区只按普通表格纵向展示地块，不符合实体大富翁地图的四周环形排版；中间区域也未用单元格铺满，视觉上不像表格游戏盘。
- 房间创建、房间列表、当前房间信息展示存在字段冗余、列宽不足、成员状态不清晰和重复入房约束不足的问题。
- 房间内缺少当前游戏展示与房主切换游戏能力，缺少房主解散房间入口。
- 战绩面板已有入口但统计数据为零值，玩家无法看到连连看、大富翁、三国杀的有效战绩。
- 连连看点到不同内容时提示“连不上”容易误导，应区分点错和路径不通。
- 大富翁对局右侧需要在日志下方补充聊天区，便于对局内交流。

一句话价值主张：让房间流转、大富翁对局和玩家资料从“能用”提升到“符合玩家预期且可验证”。

成功指标：
- 大富翁对局首屏呈现四周环形国家地块，中间网格铺满，不再是纵向列表。
- 已在房间内的账号无法再创建或加入第二个房间。
- 房间列表与当前房间页字段符合本 PRD 验收文案。
- 战绩面板能展示非零统计来源，连连看结算后资料页可见统计变化。
- 大富翁对局中日志和聊天区同时可见，聊天消息可发送并自动滚动。

## 2. 用户与场景
- 目标用户：联机房间玩家、房主、查看资料的玩家、连连看单人挑战玩家。
- 关键场景：
  - 玩家在房间列表选择游戏并创建房间。
  - 玩家进入当前房间，查看成员、当前游戏、准备并开始。
  - 房主在等待房间内切换当前游戏或解散房间。
  - 大富翁开始后，玩家在环形地图上掷骰、买地、看日志并聊天。
  - 玩家完成连连看后查看自己的战绩。

## 4. 交互设计
- 房间列表：第一行是列表表头；第二行直接提供“选择游戏”和“创建房间”操作；不再出现“创建类型”。
- 当前房间：等待态主区域是房间成员表和房间信息区；当前游戏显示在信息区。房主点击当前游戏单元格在“三国杀”和“世界版大富翁”之间切换。
- 切换游戏通知：房主切换成功后，房间聊天区新增系统消息，格式为“系统：房主已将当前游戏切换为 <游戏名>”。
- 解散房间：房主点击“解散房间”后房间关闭，所有成员回到房间列表；普通离开仍保留现有离开逻辑。
- 大富翁对局：左侧是表格化环形棋盘，中间为空白单元格区；右侧自上而下是玩家资产、日志、聊天区。
- 连连看：第二次点击不同内容时，不进行路径判定的失败文案提示，只提示选择相同内容并保留第二个格子为当前选择。

## 5. 规则与约束
- 本需求不改变三国杀出牌、伤害、响应窗口等核心规则。
- 大富翁仍为服务端权威状态，客户端只展示和发起掷骰、购买、跳过、聊天操作。
- 房间内游戏切换仅允许等待态，游戏开始后不可切换。
- 模拟测试房保留原有武将、类型、添加角色等调试能力。
- 系统通知作为房间聊天消息展示，不计入玩家战绩。

## 6. 非功能要求
- 桌面端优先，窄屏下大富翁地图和右侧栏允许滚动但不能互相遮挡。
- 单元格文字不得溢出到相邻格；长游戏名可截断但核心词可见。
- 房间状态约束由服务端二次校验，避免多标签或并发操作绕过。
- 战绩统计读取失败时资料页仍可打开，并显示零值兜底。

## 7. 产品维度自检
- [x] 清晰：每条需求均有可执行验收判据。
- [x] 完整：覆盖房间列表、当前房间、对局、资料、连连看提示、异常约束。
- [x] 价值：对应用户提出的排版、交互、约束与战绩问题。
- [x] 优先级：P0 聚焦阻断体验，P1 为增强项，P0 未超过总量一半过多的风险已接受，因为用户明确列为同批核心修复。

## 8. 评审结论回写区
- frontend-feasibility：pass。影响 `RoomListGrid`、`LobbyGrid`、`GameGrid`、`MonopolyGrid`、`LianliankanGrid`、`App`，复杂度 M，现有组件可承接。
- backend-feasibility：pass-with-conditions。需要服务端增加房间唯一性校验、房主解散/切换游戏、战绩统计来源；切换游戏限定等待态。
- qa-testability：pass。所有需求均可通过静态渲染、服务端单元测试或交互测试观察。

## 9. CR 草案
按用户图示重做大富翁棋盘为环形表格地图，补齐大富翁聊天区；修正房间列表与当前房间交互，增加房主切换游戏/解散房间和单账号单房间约束；完善分游戏战绩统计，并优化连连看不同内容误点提示。
