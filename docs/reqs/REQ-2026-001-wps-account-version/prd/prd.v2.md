## 1. 背景与目标

（同 v1，未变）项目已实现 WPS 表格伪装、房间、房间内聊天与模拟测试房完整对局，现在需要补齐账号、版本切换、大厅聊天、以及把 WPS 视觉锁死。

成功指标：

- 从首页进入登录 → 完成注册 → 进入大厅 ≤ 60 秒。
- 大厅任一 1440×900 视口截图，**非背景色像素占视口面积 ≤ 3%**（见 §4.6）。
- 玩家重连（同账号，5 分钟内）恢复成功率 ≥ 99%。
- 房间列表页显示"最近 60 秒 ≥ 1 条大厅聊天"时的挂机率显著下降。

## 4. 交互设计

### 4.1 登录 / 注册

（同 v1）以 WPS "数据校验" 对话框呈现，非全屏遮罩；登录页表格空间保持可见但不可交互。

### 4.2 修改密码

同 v1。补充：**成功后服务端使当前账号下所有 refresh token 失效**；客户端提示 `修改成功，请重新登录`，然后**广播 auth:invalidated 到该账号所有 socket**，其它标签页也会掉回登录页。

### 4.3 版本切换（**主入口 Ribbon 按钮，命令为辅**）

- **主入口**：Ribbon 增加"版本"按钮（下拉列表 + 当前项高亮）。
- **辅入口**：公式栏 `/version standard-2014` 命令。
- 切换效果：房间列表按 `version === current` 过滤；创建房间默认使用当前版本；持久化到账号偏好；同账号的其它 socket 同步更新（广播 `version:switched`）。

### 4.4 大厅聊天区

- 位置：房间列表 sheet 视图右侧固定占位（占屏幕宽度 20–25%）。
- 头部：`聊天区（大厅）`；消息从上到下追加，自动滚动到底部。
- 输入：公式栏。
- **未登录**：**只能看，不能发**；输入框显示占位符"登录后可发送消息"，Enter 提示登录。
- **进入房间后**：切换到房间频道；大厅频道停止接收。
- **老板键 Ctrl+Shift+H**：聊天区隐藏。
- **历史与限流**（服务端保证）：
  - 服务端持久化最近 **1000 条**（滚动窗口，超出淘汰，落 SQLite）。
  - 客户端建立连接后拉最近 **100 条** 快照，再订阅实时。
  - 单账号 **1 条/秒**，单条 **≤ 200 字**；超限返回错误码 `E_CHAT_RATE_LIMIT` / `E_CHAT_TOO_LONG`。

### 4.5 房间创建 / 加入

（同 v1）房间列表列：`房间号 / 状态 / 玩家人数 / 房主 / 版本 / 备注`。

### 4.6 单元格填满（R-2 判据）

- 布局定义：视口除以下**保留区**外，均属"表格区"：
  - Ribbon（顶部）
  - 公式栏
  - 状态栏
  - 聊天区（含 header）
- **判据（QA 用）**：对表格区**内部**做像素采样，**非背景色像素**（背景色由前端提供的 CSS token 定义）占**视口总面积**的比例 ≤ 3%。
- **视口分辨率基线**：1440×900、1920×1080、2560×1440 三档均通过；resize 途中允许 ≤ 500ms 的重排毛刺。
- **QA 授权**：允许 QA 建立视觉截图基线并保存到 `qa/baseline/*.png`，随需求档案一起入库。
- 反例：右下角出现纯白 / 底色不同的区域；行高抖动；聊天区与表格之间出现分割空白。

### 4.7 未登录可见范围

- **可见**：WPS 外壳（标题栏、Ribbon、公式栏、Sheet 标签、状态栏）、房间列表（只读，展示房间号/状态/人数/房主/版本/备注）、大厅聊天（只读，不可发）。
- **不可见 / 不可操作**：`创建房间`、`加入房间`、`版本切换`、发聊天、进入房间。
- **对应错误码**：`E_UNAUTHORIZED`。
- **登录对话框**：默认覆盖在房间列表 sheet 中央，允许 `Esc` 关闭（关闭后仍可只读浏览），也可点击"登录"按钮再次唤起。

### 4.8 Sandbox 测试房

- **环境开关**：由后端环境变量 `SANDBOX_ENABLED=true|false` 控制。
- **生效范围**：`ENABLE_SANDBOX=true` 时（dev / staging），`/sandbox` 命令 + Ribbon "测试房"按钮可用；生产构建默认 `false`，两个入口都不渲染。
- **登录要求**：开启时也需**登录后可进**（保持账号体系一致）。

## 5. 规则与约束（领域敏感）

### 5.1 版本切换与引擎解耦

（同 v1）本需求只影响 `Room.version` + 房间容量；不改战斗规则。

### 5.2 Session 机制（**双 token**）

- **Access Token**：JWT，TTL = **1 小时**，签名密钥后端保管；socket handshake 与 http 请求都携带。
- **Refresh Token**：不透明 token，TTL = **7 天**，仅 `/auth/refresh` 使用；每次刷新旋转（旧 refresh 立即失效）。
- **断线保坐**：进入房间/对局后断线，**5 分钟**内服务端保留玩家座位与引擎绑定；超时按现有断线策略处置（改密码时立即失效，不受 5 分钟保护）。
- **实现口径**：具体 Cookie vs Header、加密算法由 `backend-design` 在契约里定死，PRD 只约束行为。

### 5.3 服务端权威 & 反作弊

同 v1；补充：所有大厅聊天广播由服务端 fan-out，客户端不直连其他客户端。

## 6. 非功能要求

- **性能**：登录 P95 ≤ 200ms；大厅聊天端到端 ≤ 300ms；房间列表分页每页 ≤ 50 条。
- **安全**：密码 bcrypt/argon2（design 定）；登录失败限流（10 次/分/IP）；聊天长度截断；本期无过滤词库。
- **兼容**：Chrome 最近 2 个大版本 + Edge。
- **可观测**：登录/注册/改密/大厅聊天/版本切换事件均落日志。
- **可访问**：登录对话框可 tab 切换，Enter 提交，Esc 关闭（未登录首屏也可关，见 §4.7）。
- **存储选型**：**MVP 使用 SQLite**（文件放 `server/data/*.db`，纳入 `.gitignore`），生产可切 Postgres；由 `backend-design` 选 ORM 与目录布局。

## 9. CR 草案（更新版）
