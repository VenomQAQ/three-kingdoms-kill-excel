# PRD · REQ-2026-001 · WPS 摸鱼版 · 账号 + 版本切换 + 全局聊天 + 单元格填满

- **版本**：v1
- **作者**：prd-author
- **状态**：①已定稿（待②评审）
- **时间**：2026-07-02

## 1. 背景与目标

现状：项目已实现 WPS 表格伪装外壳、房间创建/加入、房间内聊天、模拟测试房完整对局流程；规则目标为《三国杀标准版·界限突破》。

痛点与目标：

- 没有账号体系，玩家昵称易撞、无法找回身份、断线重连仅按昵称匹配。
- 三国杀有多个版本（军争、SP、国战…），当前版本硬编码在 `MAX_ROOM_PLAYERS = 10`，未来扩展需要"版本目录 + 版本切换"。
- 全局聊天缺失：房间列表页只能等着看房间刷新，玩家找队友、约局需要一个"大厅聊天"。
- WPS 表格外观还有"空白区留白"：给人的第一眼不够 Excel，需要用对齐单元格填满。

一句话价值：让"摸鱼版三国杀"看起来更像正宗 Excel，同时补齐账号 + 版本能力，为后续多版本扩展打底。

成功指标（可量化）：

- 从首页进入登录 → 完成注册 → 进入大厅 ≤ 60 秒。
- 大厅任意 1440×900 视口截图，非表格空白像素占比 ≤ 3%（含 Ribbon、状态栏后）。
- 玩家重连（同账号）恢复成功率 ≥ 99%（对比现状按昵称匹配的丢座率）。
- 单个房间列表页可见"最近 60 秒" ≥ 1 条大厅聊天时的挂机率下降（对比无聊天时）。

## 2. 用户 & 场景

- **摸鱼上班族**：随时切换到"区域销售"假表；账号需要能记住昵称与偏好版本。
- **老玩家**：偏好《军争》《SP》后续版本，进入房间前先选版本。
- **新玩家**：先在大厅聊天问玩法，再决定加入哪个房间。

关键场景：

1. 陌生人首次访问：进入首页 → 看到登录/注册（WPS 登录对话框式）→ 用 QQ 邮箱注册（不发验证邮件，密码强度校验即可）→ 进入大厅。
2. 老用户登录：登录 → 恢复上次选的版本（默认《标准版·界限突破》）→ 大厅列表 + 聊天区。
3. 忘记密码/改密码：本期仅支持"登录态下修改密码"（不做邮箱找回）。
4. 版本切换：大厅右上角/公式栏 `/version` 命令，切换后房间列表按版本过滤，创建房间时按版本容量限制。
5. 大厅聊天：右侧栏 20% 宽，占满 sheet 视口高度；单元格网格延伸到聊天区之外的空白区。

## 4. 交互设计

### 4.1 登录 / 注册

在 WPS 外壳里以"数据校验"式对话框呈现（非全屏遮罩），单元格标题栏文案：`登录 三国杀·WPS 摸鱼版`。

- 未登录时首屏：默认停在"房间列表" sheet，但屏幕中央出现登录对话框；`Esc` 不允许关闭。
- 对话框字段：QQ 邮箱、密码；`注册` 与 `登录` 用同一表单，通过顶部 tab 切换。
- 校验失败：单元格红字文案 `#REF! 邮箱或密码错误`（沿用 Excel 报错风）。
- 成功后：写入 session；显示 `已登录 xxx@qq.com`（现有 InfoBar `表格用户` 位置替换）。

### 4.2 修改密码

- 入口：状态栏"表格用户 xxx@qq.com"点击 → 下拉菜单 `修改密码 / 登出`。
- 表单：原密码 / 新密码 / 确认新密码；提交成功后弹出 `修改成功，请重新登录` → 自动登出。

### 4.3 版本切换

- 入口 1：Ribbon 增加"版本"按钮（沿用 Ribbon 视觉规范：图标 + 中文名）。
- 入口 2：公式栏命令 `/version standard-2014`。
- 切换效果：房间列表按 `version === current` 过滤；创建房间默认使用当前版本；持久化到当前账号。
- **不影响已进行的对局**：进入房间后当前会话锁死为该房间的版本。

### 4.4 大厅聊天区

- 位置：房间列表 sheet 视图，右侧固定占位 `chatCols`（占位列宽由前端 design 决定，本 PRD 只说"占屏幕宽度 20%–25% 之间"）。
- 头部：`聊天区（大厅）`；下方消息列表按时间从上到下追加，新消息自动滚动到底。
- 输入：复用公式栏（原来"输入公式或 /命令"）；在大厅 sheet 下输入普通文字时消息广播到大厅频道。
- **进入房间后**：聊天区自动切换为房间频道；大厅频道不再收 / 不再显示。
- **老板键 Ctrl+Shift+H** 切"区域销售"时：聊天区隐藏。

### 4.5 房间创建 / 加入

- 创建房间：`创建房间` 按钮弹出小面板，字段：`版本`（默认当前版本）+ `房间名`（可选，默认自动）+ `密码`（可选，本期不做）→ 生成 8 位房间号。
- 加入：点击房间号，服务端校验版本容量。
- 房间列表列（在现有基础上增加"版本"列）：`房间号 / 状态 / 玩家人数 / 房主 / 版本 / 备注`。

### 4.6 单元格填满（R-2）

- 定义：从视口最左像素到最右像素、从 Ribbon 底部到状态栏顶部，除固定 UI（Ribbon / 公式栏 / 状态栏 / 聊天区）外的所有区域，必须被 `SpreadsheetGrid` 的空单元格覆盖。
- 具体要求：
  - 行高统一：与现有 `SpreadsheetGrid.module.css` 保持一致，禁止个别行拉高。
  - 列宽对齐：数据区列宽与空单元格列宽一致，不允许 `flex: 1` 造成的错位。
  - 聊天区左边界必须与某列右边界像素对齐，不出现 1–2px 缝。
  - 视口变化（resize）时，空单元格数量按视口自适应补齐/裁剪。
- 反例（判定为不通过）：右下角出现纯白 / 底色不同的区域；行高抖动；聊天区与表格之间出现分割空白。

## 5. 规则与约束（领域敏感）

- 涉及卡牌 / 技能：本需求**不改战斗规则**，出牌流程、结算队列、判定逻辑不变。
- 版本切换影响面：只影响"房间容量 + 房间列表过滤"，不改引擎行为；引擎侧沿用现有 `SangokushiEngine`。
- 服务端权威原则保留：房间创建/加入/聊天全部走服务端事件；聊天消息不由客户端广播。
- 与现有 `sandbox` 房间关系：老 `/sandbox` 命令保留（测试房不做登录强制，方便调试）；但产品可以在 PRD §6 决定"是否隐藏 sandbox 入口"（本期决定：**保留**，因为它是主要联调工具）。

## 6. 非功能要求

- **性能**：登录接口 P95 ≤ 200ms；大厅聊天单条消息端到端 ≤ 300ms；房间列表分页每页 ≤ 50 条。
- **安全**：密码 bcrypt/argon2 哈希（后端 design 定），登录失败次数限流；聊天内容做长度截断（≤ 200 字符）；本期不做过滤词库。
- **兼容**：Chrome 最近 2 个大版本 + Edge；不做 IE / 老 Safari。
- **可观测**：登录成功/失败、注册、密码修改事件都要日志；大厅聊天频道消息量按分钟聚合可查。
- **可访问**：登录对话框可键盘 tab 切换，`Enter` 提交，`Esc` 关闭（未登录首屏不可关）。
