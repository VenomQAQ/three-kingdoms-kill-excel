## 1. v1 条件复核

| v1 条件 | v2 决议 | 是否消解 | 备注 |
|---|---|---|---|
| selecting 字段落点 | 正式房使用 `room.status='selecting'` | 是 | 前端判断路径单一，房间列表可直接显示「选将中」 |
| 候选武将列表服务端权威下发 | PRD 固定服务端下发，客户端只展示/提交 | 是 | 前端不做 shuffle，不补齐候选 |
| 对局日志昵称/武将名格式 | 对局动作日志以武将名为主，其他身份场景用昵称 | 是 | `display.ts` 需拆分玩家昵称展示与日志动作展示 |

## 2. 新增/修订项影响

| 项 | 前端影响 | 结论 |
|---|---|---|
| `room:join` 复用重进 | sessionStorage 恢复后调用现有加入动作；需识别本人房间「返回」文案 | 可行 |
| 选将 UI | 新增/扩展选将面板，候选卡展示武将名、技能、体力/血量、剩余时间 | 可行 |
| 180s 超时 | 展示倒计时；超时动作由服务端权威执行，前端只同步状态 | 可行 |
| 测试房保留原逻辑 | 前端保持 sandbox 快捷入口与自定义角色流程，不接正式房选将 UI 强制分支 | 可行 |
| 昵称替换 | 首页、房间列表、聊天、大厅、状态栏等替换邮箱展示 | 可行 |
| 对局日志武将名主展示 | 日志渲染层优先使用 generalName；必要时显示 `武将名（昵称）` | 可行 |
| 对局区铺满 | 选将 Sheet 纳入 game-sheet 像素基线 | 可行 |

## 3. 成本 & 风险

- 更新后粗估：**5–8 人日**，与 v1 基本一致。
- 最大增量仍是正式房选将 UI；v2 明确字段、倒计时和候选信息后，前端无产品阻断。
- 展示规则需要避免误伤账号表单：邮箱只在登录/注册/改密等账号场景显示，其它玩家可见场景用昵称。

## 4. 判定

**pass**：v1 前端条件已由 PRD v2 消解；新增的 180s、测试房保留原逻辑、日志武将名主展示均可在现有 React/store 结构中实现。可进入 design。
